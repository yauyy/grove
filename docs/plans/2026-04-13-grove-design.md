# Grove - Multi-Project Git Worktree Workspace Manager

> A cross-platform (macOS/Windows) Rust CLI tool for managing git worktrees across multiple projects.

## Overview

Grove lets you register multiple local git projects, group them, and create unified workspaces where each project gets its own worktree branch. It provides batch git operations across all projects in a workspace, making multi-repo development workflows seamless.

---

## 1. Command Reference

### Project Management

| Command | Shortcut | Description |
|---------|----------|-------------|
| `grove add <path>` | | Register a local git project |
| `grove remove` | | Interactively remove a registered project |
| `grove list` | `grove ls` | List all registered projects (grouped) |

### Group Management

| Command | Description |
|---------|-------------|
| `grove group add <name>` | Create a group |
| `grove group remove` | Interactively remove a group (projects become "ungrouped") |
| `grove group list` | List all groups and their projects |
| `grove group reorder` | Interactively reorder groups |
| `grove move <project>` | Move a project between groups or reorder |

### Workspace Management

| Command | Shortcut | Description |
|---------|----------|-------------|
| `grove create [name]` | `grove -c [name]` | Create a new workspace |
| `grove -w [name]` | | Edit existing workspace (add/remove projects) |
| `grove delete` | | Delete a workspace (cleanup worktrees + branches) |
| `grove status` | `grove st` | View all workspaces and their git status |

### Configuration

| Command | Description |
|---------|-------------|
| `grove config set workpath <path>` | Set workspace root path (affects new workspaces only) |
| `grove config list` | Show current configuration |
| `grove completion <shell>` | Generate shell completion script (zsh/bash/fish/powershell) |
| `grove help` | Show help (auto-generated by clap) |

### Batch Git Operations

All batch git commands operate on the current workspace (auto-detected from cwd, or interactive selection).

| Command | Shortcut | Description |
|---------|----------|-------------|
| `grove sync` | `grove sy` | fetch + merge remote main branch into work branch |
| `grove gmerge` | `grove gm` | Merge work branch -> environment branch (interactive select) |
| `grove gstatus` | `grove gs` | git status for all projects |
| `grove gadd` | `grove ga` | git add . for all projects |
| `grove gcommit` | `grove gc` | git commit with unified message for all projects |
| `grove gpush` | `grove gp` | git push -u origin \<branch\> for all projects |
| `grove gpull` | `grove gl` | git pull for all projects |

## 2. Configuration

### Directory Structure

```
~/.grove/
├── config.toml          # Global config
├── projects.toml        # Registered projects
├── workspaces.toml      # Created workspace records
└── agents/              # Per-project agents.md files
    ├── project-a.md
    └── project-b.md
```

### config.toml

```toml
workpath = "~/grove-workspaces"
```

**Default workpath:**
- macOS: `~/grove-workspaces`
- Windows: `C:\Users\<user>\grove-workspaces`

Changes to `workpath` only affect **newly created** workspaces. Existing workspaces are not affected.

### projects.toml

```toml
[[groups]]
name = "Frontend"
order = 1

[[groups]]
name = "Backend"
order = 2

[[projects]]
name = "web-app"
path = "/Users/yau/projects/web-app"
group = "Frontend"
order = 1
agents_md = "~/.grove/agents/web-app.md"  # optional

[projects.branches]
main = "origin/master"        # Remote main branch (required)
test = "origin/test"          # Test environment (optional)
staging = "origin/staging"    # Staging environment (optional)
prod = "origin/main"          # Production environment (optional)

[[projects]]
name = "api-server"
path = "/Users/yau/projects/api-server"
group = "Backend"
order = 1

[projects.branches]
main = "origin/main"
test = "origin/test"
```

### workspaces.toml

```toml
[[workspaces]]
name = "feature-login"
branch = "feature-login"
created_at = "2026-04-13"

[[workspaces.projects]]
name = "web-app"
worktree_path = "~/grove-workspaces/feature-login/web-app"

[[workspaces.projects]]
name = "api-server"
worktree_path = "~/grove-workspaces/feature-login/api-server"
```

### Workspace Directory Layout

```
~/grove-workspaces/feature-login/
├── AGENTS.md              # Merged agents.md from all selected projects
├── web-app/               # git worktree
└── api-server/            # git worktree
```

## 3. Core Flows

### `grove add <path>`

```
1. Validate path is a valid git repository
2. Auto-detect project name (directory name), allow user to modify
3. Interactive: select group (existing groups + "New group" + "Ungrouped")
4. Interactive: select remote main branch
   > Lists all remote branches for selection (e.g., origin/master, origin/main)
5. Interactive: configure environment branches (each optional, skip with empty Enter)
   > Test branch? [skip]: origin/test
   > Staging branch? [skip]: origin/staging
   > Production branch? [skip]: origin/prod
6. Interactive: configure agents.md? (optional)
   > If yes: enter path or open editor
7. Save to projects.toml
```

Path supports both absolute and relative paths; stored as absolute path internally.

### `grove create [name]`

```
1. If no name provided: prompt for workspace name
2. Display registered projects grouped by group, space to multi-select, Enter to confirm
3. Prompt for branch name:
   > Branch name [feature-login]:    <- placeholder is workspace name
   > Empty Enter = use workspace name as branch name
4. Create workspace directory: ~/grove-workspaces/feature-login/
5. For each selected project:
   a. cd to project's original repo path
   b. git fetch origin
   c. git worktree add <workspace-path>/<project-name> -b <branch> --no-track <main-branch>
      (Based on remote main branch, NO auto-tracking)
   d. If one project fails: report error, continue with others
6. Merge all selected projects' agents.md -> <workspace>/AGENTS.md
7. Save workspace info to workspaces.toml
8. Print creation summary
```

### `grove -w [name]` (Edit Workspace)

```
1. If no name: Tab-cycle through existing workspaces
   > Tab = next workspace, cycles back to first at the end
2. Enter edit mode:
   > Display all registered projects (reuse create's multi-select UI)
   > Projects already in workspace are pre-checked ✓
   > Space to toggle, Enter to confirm
3. Process changes:
   > Newly checked projects: create worktree (same branch, --no-track)
   > Unchecked projects: check for uncommitted changes
     - Has uncommitted changes -> ERROR, abort operation
     - No uncommitted changes -> git worktree remove
4. Regenerate AGENTS.md from current project set
5. Update workspaces.toml
```

### `grove gmerge`

```
1. Detect current workspace (from cwd, or interactive selection)
2. Collect environment branch configs from all projects, compute intersection
   > If intersection is empty: "No common environment configured across all projects"
   > If a project is missing one: "Project X has not configured Y environment"
3. Interactive: single-select target environment (only show intersection)
4. For each project:
   a. git fetch origin
   b. git checkout <env-branch>    (local tracking branch of env)
   c. git merge <work-branch>
   d. git checkout <work-branch>   (switch back)
5. Report results per project (success / conflict / failure)
```

### `grove sync`

```
1. Detect current workspace
2. For each project:
   a. git fetch origin
   b. git merge <remote-main-branch>   (e.g., origin/master)
3. Report results per project
```

### `grove delete`

```
1. Interactive: select workspace to delete
2. Check all projects for uncommitted changes
   > If any: warn and require confirmation
3. For each project:
   a. git worktree remove <worktree-path>
   b. git branch -d <branch>   (delete local branch)
4. Remove workspace directory
5. Remove entry from workspaces.toml
```

## 4. Batch Git Operations Detail

All batch operations follow the **continue-and-summarize** error strategy:

```
grove gpush

  ✓ web-app        -> origin/feature-login (pushed)
  ✗ api-server     -> Push failed: rejected (non-fast-forward)
  ✓ common-lib     -> origin/feature-login (pushed)

Result: 2 succeeded, 1 failed
```

### gadd
- Runs `git add .` in each project's worktree directory

### gcommit
- Prompts for a single commit message
- Runs `git commit -m "<message>"` in each project's worktree

### gpush
- Runs `git push -u origin <branch>` (push to same-name remote branch)
- Reports per-project success/failure with specific error

### gpull
- Runs `git pull` in each project's worktree

### gstatus
- Runs `git status --short` in each project's worktree
- Displays grouped output per project

### Workspace Detection

Batch git commands need to know "which workspace am I in":
1. Get current working directory
2. Iterate workspaces in workspaces.toml
3. Check if cwd is under any workspace path
4. If not in any workspace: interactive selection

## 5. Error Handling & Edge Cases

| Scenario | Handling |
|----------|----------|
| `add` path is not a git repo | Error: "Not a valid git repository" |
| `add` project already registered | Error: "Project already registered, use grove list" |
| `create` workspace name exists | Error: "Workspace name already exists" |
| `create` branch exists in a project | Error: name which project, offer rename or skip |
| `gmerge` merge conflict | Report conflicting files, prompt user to resolve manually |
| `delete` with uncommitted changes | Warn and require confirmation |
| Not in any workspace (batch ops) | Interactive workspace selection |
| `config set workpath` new path missing | Auto-create directory |
| Remote branch doesn't exist | Check after fetch, give clear message |
| Windows path separators | Use `std::path::PathBuf` throughout |

Generated via `clap_complete`:

```bash
# zsh
grove completion zsh > _grove

# bash
grove completion bash > grove.bash

# fish
grove completion fish > grove.fish

grove completion powershell > _grove.ps1
```

Completions include:
- All subcommands and flags
- Workspace names after `-w`
- File paths after `add`
- Group names after `group` subcommands

## 7. Tech Stack

| Purpose | Crate | Rationale |
|---------|-------|-----------|
| CLI framework | `clap` (derive) | Arg parsing, auto help, shell completions |
| Terminal interaction | `dialoguer` | Multi-select, single-select, input with placeholder |
| Terminal styling | `console` | Colors, progress (dialoguer dependency) |
| Config serialization | `serde` + `toml` | TOML read/write |
| Git operations | `std::process::Command` | Direct git CLI calls (worktree support in git2/libgit2 is incomplete) |
| Path handling | `dirs` | Cross-platform home directory |
| Error handling | `anyhow` | Simplified error chains |
| Date/time | `chrono` | Workspace creation timestamps |

**Prerequisite:** Git must be installed on the user's system.

## 9. Key Design Principles

1. **No auto-tracking**: All worktrees created with `--no-track` (`git switch -c <branch> --no-track <remote/main>`)
2. **Continue-and-summarize**: Batch operations don't stop on first failure; report all results
3. **Minimal typing**: Short aliases for all frequent commands, Tab completion
4. **Config changes are forward-only**: `config set workpath` only affects new workspaces
5. **Cross-platform**: Use `std::path::PathBuf`, `dirs` crate, test on macOS + Windows
6. **AGENTS.md auto-merge**: Regenerated whenever workspace project set changes (create, -w edit)
